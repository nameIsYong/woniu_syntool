# 档案合并功能知识库

> 本知识库用于 AI / 智能体开发参考，涵盖档案合并功能的所有 API 接口、数据模型、枚举映射、业务规则等核心信息。
> 对应需求文档：`档案合并需求文档.md`

---

## 1. 功能概述

档案合并用于解决两份居民健康档案数据合并时的冲突处理。

**核心流程**：
1. 登录获取 Token
2. 搜索并选择档案 A 和档案 B（不能是同一份）
3. 指定主/辅档案关系
4. 系统自动对齐数据，标记冲突项
5. 用户手动解决冲突
6. 构造保存参数并提交

---

## 2. API 接口清单

### 2.1 登录认证

复用项目公共登录能力 `AuthService.login(account, password)`。

- **端点**：`https://wndl.2woniu.cn/sign/sso/pass`
- **返回**：`token`、`authToken`、`institutionName`、`account`

> 注意：档案合并页面的登录信息**禁止数据持久化**，退出页面即失效。

### 2.2 搜索档案列表

```
GET http://bmg.2woniu.cn/bmanage/publichealth/healthrecord/list
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| unRegionCode | int | 是 | 固定为 0 |
| keyword | string | 是 | 搜索关键词，支持姓名、身份证号 |
| pageNo | int | 是 | 分页页码，从 1 开始 |
| pageSize | int | 是 | 固定为 10 |
| status | int | 是 | 档案状态（见枚举 `archivesStatusOptions`） |
| token | string | 是 | Header 中传递 |

**返回结构**：

```json

  "data": {
    "pageSize": 10,
    "pageNo": 1,
    "total": 1,
    "results": [

        "name": "周宁近",
        "phoneNumber": "无",
```

### 2.3 查询档案详情

```
GET http://wnjk.2woniu.cn/wnjkapp/resident/archives/detail
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| residentHealthRecordId | string | 是 | 档案 ID |
| token | string | 是 | Header 中传递 |

**返回结构**：

```json
{
  "status": 0,
  "msg": "操作成功",
  "data": {
    "yptData": {
      "familyInfo": { ... },
      "personBaseInfo": { ... },
      "personBaseExtendsInfo": { ... },
      "personHistoryList": [ ... ],
      "personFamilyHistoryList": [ ... ],
      "personIllnessList": [ ... ],
      "livingEnvironment": { ... }

```

## 5. 位运算枚举处理

对于位运算组合值的枚举字段，需要将数值拆分为二进制位，把每一位对应的枚举名称用逗号连接显示。

**辅助函数**：

```dart
/// 位运算枚举值转中文名称
/// 例如 value=532, map={1:'无',2:'高血压',4:'糖尿病'} -> '高血压,糖尿病,慢性阻塞性肺疾病,肝炎'
String bitEnumToString(int? value, Map<int, String> enumMap) {
  if (value == null || value == 0) return '';
  final List<String> results = [];
  enumMap.forEach((key, label) {
    if (key != 0 && (value & key) == key) {
      results.add(label);
    }
  });
  return results.join(', ');
}
```

## 6. 合并规则

### 6.1 字段存在性定义

- **不存在（视为空）**：
  - 字段缺失（JSON 中无此 key，或值为 `null`）
  - 字符串类型值为空字符串 `""`
  - 数组类型值为空数组 `[]`
  - 对象类型值为空对象 `{}`（所有子字段均视为空）
- **存在**：字段有值且不满足上述空值条件。

### 6.2 基础对象字段合并（familyInfo / personBaseInfo / livingEnvironment）

将三个对象的所有叶子字段展开为独立的合并项，逐字段对比。

| 类型 | 定义 | 处理方式 | 卡片色调 |
|------|------|----------|----------|
| 两边都不存在 | 主辅两边该字段均为空 | **隐藏**，不显示 | - |
| 仅主档案存在 | 主档案有效，辅档案为空 | **自动保留主档案值** | 白色 |
| 仅辅档案存在 | 辅档案有效，主档案为空 | **自动保留辅档案值（新增）** | 绿色 |
| 两边都存在且值相等 | 主辅两边均有效且值一致 | **自动保留主档案值** | 白色 |
| 两边都存在且值不等 | 主辅两边均有效但值不一致 | **需用户手动选择** | 红色（未解决）→ 绿色（已解决） |

**值相等判断**：对于基础类型（String、int、double），直接比较 `toString()` 是否相等。对于布尔类型，直接比较值。

### 6.3 数组类型字段合并

档案模型中有三个数组字段：`personHistoryList`、`personFamilyHistoryList`、`personIllnessList`。

**数组存在性定义**：
- 数组为 `null`、空数组 `[]`、或字段缺失 → 视为空。

**数组冲突判定规则**：

1. **主数组为空，辅数组不为空**（至少有一个元素） → **新增**（绿色）。
2. **两边数组元素个数不一致** → **冲突**（红色）。
3. **两边数组元素个数一致** → 需要进一步按多字段组合判断（注意数组元素可能是乱序）：
   - **PersonHistory**：通过 `recordType + occurrenceDate + name + occurrenceDateOther` 组合判断是否相同。
   - **PersonFamilyHistory**：通过 `disease + relationshipType + remark` 组合判断是否相同。
   - **PersonIllness**：通过 `diseaseKindId + remark + diagnosisDate + recordDate + diagnosisDateOther` 组合判断是否相同。
   
   逐条匹配：若主档案的每条记录都能在辅档案中找到一条完全匹配的记录，且辅档案没有多余记录，则视为**两边相等**（自动保留主）。否则视为**冲突**。

4. **数组冲突的解决方式**：用户选择"保留主"或"保留辅"，即**整个数组替换**。

### 6.4 扩展数据 personBaseExtendsInfo

- **不参与合并展示**，不在冲突列表中显示。
- 保存时需要按规则构造（见第 7.3 节）。

## 7. 保存规则

### 7.1 保存参数构造

以**主档案**为基础，按以下规则构造保存参数：

1. **基础对象字段（`familyInfo`、`personBaseInfo`、`livingEnvironment`）**：
   - 仅主档案存在 / 两边相等 → 保留主档案值。

   - 新增 → 保留辅档案数组（若未取消）。
   - 两边相等 / 仅主存在 → 保留主档案数组。

   - 不将对象本身作为一条计数，而是递归统计其内部所有叶子字段（非对象、非数组的基础字段）中值为"存在"的数量。

2. **数组字段（`personHistoryList`、`personFamilyHistoryList`、`personIllnessList`）**：
   - 统计数组中的元素个数（每个元素算 1 条）。
   - 空数组计 0 条。

3. **不参与统计的字段**：
   - `personBaseExtendsInfo` 不参与合并，也不计入完善数据条数。

**推荐原则**：展示档案 A 和档案 B 各自的已完善数据条数，推荐用户把完善数量少的档案合并到完善数量多的档案上（即完善少的为辅，完善多的为主）。

## 10. 排序规则

列表按以下优先级排序（从高到低）：

1. **冲突项**（未解决红色在前，已解决绿色在后）
2. **辅档案独有（新增）**（绿色）
3. **主档案独有 + 两边相等**（白色）

同一优先级内，按字段所属模块和字段名排序：家庭信息 → 居民基本信息 → 生活环境 → 既往史 → 家族史 → 疾病史。
