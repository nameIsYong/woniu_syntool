# 档案合并功能需求文档

## 1. 概述

本功能旨在解决两份居民健康档案数据合并时的冲突处理问题。用户通过关键词搜索并选择两份档案（档案A、档案B），指定主/辅关系后，系统自动对齐两份档案数据，对无冲突数据自动处理，对有冲突或仅辅档案存在的数据提供界面供用户人工决策，最终生成一份清洗后的完整档案并提交保存。

## 2. 数据基础

### 2.1 输入数据

输入为两份完整的档案详情数据，通过档案详情接口获取，使用 `ArchivesModel` 模型接收。

```
ArchivesModel
├── familyInfo: FamilyBaseInfoModel        // 家庭信息（对象）
├── personBaseInfo: PersonBaseInfo         // 居民基本信息（对象）
├── personHistoryList: [PersonHistory]     // 既往史（数组）
├── personFamilyHistoryList: [PersonFamilyHistory]  // 家族史（数组）
├── personIllnessList: [PersonIllness]     // 遗传疾病史及残疾情况（数组）
├── livingEnvironment: LivingEnvironment   // 生活环境（对象）
└── personBaseExtendsInfo: [String:Any]    // 扩展数据（字典，不参与合并展示）
```

### 2.2 字段存在性定义

- **不存在（视为空）**：
  - 字段缺失（JSON 中无此 key，或值为 `null` / `nil`）
  - 字符串类型值为空字符串 `""`
  - 数组类型值为空数组 `[]`
  - 对象类型值为空对象 `{}`（所有子字段均视为空）
- **存在**：字段有值且不满足上述空值条件。

### 2.3 已完善数据条数统计规则

用于主/辅档案推荐，统计逻辑如下：

1. **对象字段（`familyInfo`、`personBaseInfo`、`livingEnvironment`）**：
   - 不将对象本身作为一条计数，而是递归统计其内部所有叶子字段（非对象、非数组的基础字段）中值为"存在"的数量。
   - 例如 `familyInfo.houseType = "1"` 算 1 条，`familyInfo.remark = ""` 不算。

2. **数组字段（`personHistoryList`、`personFamilyHistoryList`、`personIllnessList`）**：
   - 统计数组中的元素个数（每个元素算 1 条）。
   - 空数组计 0 条。

3. **不参与统计的字段**：
   - `personBaseExtendsInfo` 不参与合并，也不计入完善数据条数。

> **主辅推荐原则**：展示档案A和档案B各自的已完善数据条数，推荐用户把完善数量少的档案合并到完善数量多的档案上（即完善少的为辅，完善多的为主）。

## 3. 页面流程

```
进入页面 → 检查登录状态（未登录弹登录框）
    ↓
搜索并选择档案A（弹框关键词搜索 → 单选）
    ↓
搜索并选择档案B（弹框关键词搜索 → 单选，校验不能与A同ID）
    ↓
查询两份档案详情 → 展示档案A/B摘要卡片 → 用户单选主档案
    ↓
系统自动计算合并 → 展示合并列表（冲突/新增/主数据）
    ↓
用户手动解决冲突 → 点击保存 → 校验冲突全部解决 → 构造保存参数并打印JSON
```

## 5. 搜索选择档案

### 5.1 搜索入口

页面上提供两个独立的搜索入口，分别用于选择**档案A**和**档案B**：

```
┌─────────────────────────────────────────────┐
│  [搜索档案A]        [搜索档案B]              │
│  姓名: 张三         姓名: 李四               │
│  身份证: 510...     身份证: 510...           │
└─────────────────────────────────────────────┘
```

- 每个搜索入口包含：关键词输入框 + 档案状态下拉框 + 搜索按钮。
- 点击搜索后弹出档案列表弹框，用户单选一条档案。
- 已选择的档案支持重新选择。
- **档案A和档案B禁止为同一条数据**，校验规则为档案ID（`residentHealthRecordId`）是否相同。

| 迁出 | 1 |
| 死亡 | 2 |
| 其他 | 3 |
| 失访 | 4 |
| 注销 | 5 |
| 删除 | 99 |

- 默认选中"正常"（`status=0`）。

## 6. 主辅档案选择

### 6.1 选择入口

查询两份档案详情后，页面以**卡片形式**展示档案A和档案B的关键信息：

```
┌─────────────────────┐    ┌─────────────────────┐
│  [ ] 档案A          │    │  [ ] 档案B          │
│  姓名: 张三          │    │  姓名: 张三(同名)    │
│  身份证: 510...      │    │  身份证: 510...      │
│  已完善: 45 条       │    │  已完善: 32 条       │
└─────────────────────┘    └─────────────────────┘
```

- 卡片支持**单选**，选中后代表以对应档案为**主档案**，另一份自动为辅档案。
- 展示两份档案各自的**已完善数据条数**（统计规则见 2.3）。
- **推荐文案**：若A完善45条、B完善32条，则在B卡片旁显示"推荐合并到A"的提示。

### 6.2 主辅切换

- 支持重新选择主/辅关系。
- 切换主辅后，所有用户已做的冲突解决选择**全部重置**，重新计算合并列表。

## 7. 合并规则

### 7.1 基础对象字段合并（`familyInfo` / `personBaseInfo` / `livingEnvironment`）

将三个对象的所有叶子字段展开为独立的合并项，逐字段对比。

**字段分类规则**：

| 类型 | 定义 | 处理方式 | 卡片色调 |
|------|------|----------|----------|
| **两边都不存在** | 主辅两边该字段均为空 | **隐藏**，不显示 | - |
| **仅主档案存在** | 主档案有效，辅档案为空 | **自动保留主档案值** | 白色 |
| **仅辅档案存在** | 辅档案有效，主档案为空 | **自动保留辅档案值（新增）** | 绿色 |
| **两边都存在且值相等** | 主辅两边均有效且值一致 | **自动保留主档案值** | 白色 |
| **两边都存在且值不等** | 主辅两边均有效但值不一致 | **需用户手动选择** | 红色（未解决）→ 绿色（已解决） |

> **值相等判断**：对于基础类型（String、int、double），直接比较 `toString()` 是否相等。对于布尔类型，直接比较值。

### 7.2 数组类型字段合并

档案模型中有三个数组字段：`personHistoryList`、`personFamilyHistoryList`、`personIllnessList`。

**数组存在性定义**：
- 数组为 `null`、空数组 `[]`、或字段缺失 → 视为空。

**数组冲突判定规则**：

1. **主数组为空，辅数组不为空**（至少有一个元素） → **新增**（绿色）。
2. **两边数组元素个数不一致** → **冲突**（红色）。
3. **两边数组元素个数一致** → 需要进一步按多字段组合判断（注意数组元素可能是乱序）：

   - **PersonHistory**：通过 `recordType + occurrenceDate + name + occurrenceDateOther` 组合判断是否相同。
   - **PersonFamilyHistory**：通过 `disease + relationshipType + remark` 组合判断是否相同。
   - **PersonIllness**：通过 `diseaseKindId + remark + diagnosisDate + recordDate + diagnosisDateOther` 组合判断是否相同。

   逐条匹配：若主档案的每条记录都能在辅档案中找到一条完全匹配的记录（基于上述字段组合），且辅档案没有多余记录，则视为**两边相等**（自动保留主）。否则视为**冲突**。

4. **数组冲突的解决方式**：用户选择"保留主"或"保留辅"，即**整个数组替换**。

**数组字段的展示**：

冲突或新增状态下，卡片中需要展示两边数组的摘要信息，便于用户对比：

```
[冲突] 遗传疾病史及残疾情况
主(2条):
  - [高血压] 确诊: 2026-04-20 建档: 2026-04-21
  - [糖尿病] 确诊: 2026-04-20 建档: 2026-04-21
辅(1条):
  - [高血压] 确诊: 2026-04-18 建档: 2026-04-19
```

### 7.3 扩展数据 `personBaseExtendsInfo`

- **不参与合并展示**，不在冲突列表中显示。
- 保存时需要按规则构造（见第 10.3 节）。

## 8. 界面显示规则

### 8.1 布局方式

**单列表 + 分组标题 + 筛选栏**

- 页面只显示一个合并后的列表。
- 列表按模块分组展示，分组标题为：
  - 家庭信息（`familyInfo`）
  - 居民基本信息（`personBaseInfo`）
  - 生活环境（`livingEnvironment`）
  - 既往史（`personHistoryList`）
  - 家族史（`personFamilyHistoryList`）
  - 遗传疾病史及残疾情况（`personIllnessList`）

### 8.2 筛选栏

列表顶部显示筛选标签，快速过滤查看不同类型的数据：

| 筛选标签 | 显示内容 |
|----------|----------|
| **全部** | 显示所有字段（除两边都不存在的） |
| **冲突** | 仅显示冲突项（未解决红色 + 已解决绿色） |
| **新增** | 仅显示辅档案独有项（绿色） |
| **主数据** | 仅显示主档案独有 + 两边相等项（白色） |

### 8.3 卡片设计

#### 主档案独有 / 两边相等（白色卡片）

```
┌──────────────────────────────────────────────┐
│ [家庭信息] 家庭地址                            │
│ 值: 四川省广安市邻水县坛同镇蓼叶寺村委会01组    │
└──────────────────────────────────────────────┘
```

#### 辅档案独有（新增，绿色卡片）

```
┌──────────────────────────────────────────────┐  ← 绿色边框/浅绿背景
│ [新增] [居民基本信息] 工作单位                  │
│ 值: 测试单位                                  │
│                                    [取消新增] │
└──────────────────────────────────────────────┘
```

- 新增项支持"取消新增"操作，取消后该字段在保存时不被包含（恢复为主档案的空状态）。
- 已取消的新增项显示为灰色，支持"恢复新增"。

#### 冲突项（红色卡片 → 解决后绿色）

**未解决状态**：

```
┌──────────────────────────────────────────────┐  ← 红色边框/浅红背景
│ [冲突] [居民基本信息] 性别                     │
│                                              │
│ ○ 主: 男 (1)                                 │
│ ○ 辅: 女 (2)                                 │
└──────────────────────────────────────────────┘
```

**已解决状态**：

```
┌──────────────────────────────────────────────┐  ← 绿色边框
│ [冲突] [居民基本信息] 性别                     │
│ ✓ 已选主                                     │
│ 值: 男                                       │
└──────────────────────────────────────────────┘
```

- 冲突项始终显示两个选项（主/辅），支持反复切换。
- 点击选择后卡片变为绿色，显示"已选主"或"已选辅"。

| recordType | 1: 手术, 2: 外伤, 3: 输血, 4: 遗传病史 |

**遗传病既往史/家族史（PersonFamilyHistory）**：

| 字段 | 枚举映射 |
|------|---------|
| relationshipType | 1: 父亲, 2: 母亲, 3: 兄弟姐妹, 4: 子女 |
| disease | 位运算组合值：1: 无, 2: 高血压, 4: 糖尿病, 8: 冠心病, 16: 慢性阻塞性肺疾病, 32: 恶性肿瘤, 64: 脑卒中, 128: 重性精神病, 256: 结核病, 512: 肝炎, 1024: 先天畸形, 2048: 其它 |

**遗传疾病史及残疾情况（PersonIllness）**：

| 字段 | 枚举映射 |
|------|---------|
| status | 0: 患病, 1: 治愈 |
| diseaseId | `52a03287-4091-4bce-86ed-10a4d2521816`: 高血压, `7aa0aff1-dda0-40f1-8510-513de3753547`: 糖尿病, `708de718-84c6-45d0-aef8-02e06c6b4b12`: 冠心病, `c2b2cb08-03ec-46a6-84a0-0ae350a34251`: 慢性阻塞性肺疾病, `e63f8cf3-00d4-4fec-bd6f-f96b9fcaf39c`: 恶性肿瘤, `37a51584-e2ef-43d0-9cef-1ce8d69bdeb8`: 脑卒中, `fc6ff7e2-7f06-4d9c-98dc-d5c83e70f119`: 严重精神障碍, `52a03287-4091-4bce-86ed-10a4d2521814`: 结核病, `ab9e834e-dc1b-4fe6-8cd4-1c94767afd9a`: 肝炎, `de414310-bbca-4729-9208-fd524d3b617a`: 其他法定传染病, `930faeaf-1da3-4246-a2cb-dda86ee480cd`: 职业病 |

> **位运算枚举显示**：对于 `disease`（残疾情况、家族史疾病等）这类位运算组合值，需要将数值拆分为二进制位，把每一位对应的枚举名称用逗号连接显示。例如 `disease = 532 = 4+8+16+512`，显示为"糖尿病, 冠心病, 慢性阻塞性肺疾病, 肝炎"。

### 8.5 排序规则

列表按以下优先级排序（从高到低）：

1. **冲突项**（未解决红色在前，已解决绿色在后）
2. **辅档案独有（新增）**（绿色）
3. **主档案独有 + 两边相等**（白色）

同一优先级内，按字段所属模块和字段名排序：家庭信息 → 居民基本信息 → 生活环境 → 既往史 → 家族史 → 疾病史。

## 9. 用户操作交互

### 9.1 冲突解决

- 冲突数据卡片显示两个选项（主/辅）供选择。

- 页面底部有**保存按钮**。

  1. 是否已登录
  2. 是否已选择主档案

- 如有未解决的冲突，给予提示不予提交。
- 校验通过后，**先打印合并后的 JSON 到控制台**（暂不真实提交），待确认无误后再启用真实保存。

### 10.1 保存接口

```
POST https://jk.test.2woniu.cn:6968/wnjkapp/resident/archives/saveOrUpdate
```

请求 Header：

| Header | 值 |
|--------|-----|
| Content-Type | application/json |
| token | 登录获取的 token |
| PP-User-Agent | os=2;ver=3.0.15;ctype=2;imei=FFC755B2-25A8-467A-879C-71592B9C7866 |

> **注意**：当前阶段先不调用真实接口，仅在前端打印构造好的 JSON 参数。确认无误后再启用真实提交。

### 10.2 保存参数构造

以**主档案**为基础，按以下规则构造保存参数：

1. **基础对象字段（`familyInfo`、`personBaseInfo`、`livingEnvironment`）**：
   - 仅主档案存在 / 两边相等 → 保留主档案值。
   - 辅档案独有（新增）→ 保留辅档案值（若用户未取消新增）。
   - 冲突已解决 → 保留用户选择的值（主或辅）。
   - 两边都不存在 / 新增已取消 → 字段值为空或按后端要求处理。

2. **数组字段（`personHistoryList`、`personFamilyHistoryList`、`personIllnessList`）**：
   - 冲突已解决 → 保留用户选择的整个数组（主或辅）。
   - 新增 → 保留辅档案数组（若未取消）。
   - 两边相等 / 仅主存在 → 保留主档案数组。

3. **固定附加字段**：
   - `callback`: `"archiveSave_APPinjection_callback"`
   - `dataSourceFrom`: `"云平台"`
   - `spkgId`: 需要从服务包接口获取（参考体检合并的 `fetchServicePackageId` 能力，复用 `http://wnjk.2woniu.cn/wnjkapp/spkg/cache` 接口）。

4. **`personBaseExtendsInfo`**：
   - 不参与合并，但需要根据规则补全默认值（见 10.3）。
