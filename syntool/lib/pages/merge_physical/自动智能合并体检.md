# 自动智能合并体检需求文档

## 1. 概述

在现有“体检合并”页面基础上，新增“智能合并”能力，用于对左侧重复体检档案列表执行批量自动合并。  
该能力应复用现有“去处理 -> 查询体检详情 -> 生成合并结果 -> 保存”的主流程，但取消人工选择主辅数据、取消人工处理冲突，改为系统按既定规则自动完成。

本需求目标：

- 降低重复体检人工逐条处理成本
- 在不改变现有手工合并能力的前提下，新增一套可批量执行的自动合并流程
- 对批量执行过程中的成功、失败、删除失败、处理进度进行可视化反馈

## 2. 页面改动

### 2.1 左侧列表区域新增按钮

在左侧“重复体检档案列表”底部新增按钮：

- 按钮名称：`智能合并`

点击后弹出“合并策略”弹框，不直接开始执行。

### 2.2 智能合并进度面板

页面需要新增“智能合并任务面板”或等价的进度展示区域，用于展示批量执行状态。建议展示以下内容：

- 总人数
- 已处理人数
- 成功人数
- 失败人数
- 删除失败人数
- 当前处理人员姓名
- 当前处理人员身份证号
- 当前步骤
- 是否已完成

其中“当前步骤”建议至少包含：

- 查询体检列表
- 查询体检详情
- 自动判定主辅数据
- 自动生成合并结果
- 保存合并结果
- 删除辅数据

页面还需要支持查看失败明细与删除失败明细，便于用户后续人工处理。

### 2.3 智能合并入口限制提示

页面需要在智能合并执行前增加必要的风控校验与提示：

- 若当前筛选日期跨度大于 180 天，则禁止使用智能合并
- 禁止时需要明确提醒用户：筛选时间跨度过大，可能误将历史正常体检数据进行错误合并

该限制仅针对“智能合并”功能，不影响现有手工“去处理”流程。

## 3. 合并策略弹框

### 3.1 弹框目标

点击“智能合并”后，需要弹出“合并策略”弹框，让用户先理解自动合并规则，再决定是否执行。

### 3.2 弹框内容

弹框中需要明确说明以下规则：

1. 系统会按左侧重复档案列表顺序，逐人串行处理
2. 每个人处理完成后才会进入下一个人
3. 主数据由系统自动判定，不需要用户手工选择
4. 冲突项自动以主数据为准，辅数据冲突值将被舍弃
5. 某个人处理失败不会中断整体流程，系统会记录失败并继续处理下一个人

### 3.3 用户可配置项

弹框中至少提供以下选项：

- `合并保存成功后，自动删除辅数据`

该选项为布尔开关，由用户勾选或取消。

### 3.4 删除接口说明

删除辅数据接口当前可先使用假接口预留，要求：

- 保留明确的方法入口
- 保留请求参数结构设计
- 返回结果结构需支持成功、失败、错误信息
- 后续替换真实接口时，不影响智能合并主流程设计

## 4. 智能合并整体流程

### 4.1 总体执行原则

智能合并必须按左侧档案列表顺序串行执行，不允许并发处理多个人员。

执行顺序：

1. 遍历左侧重复档案分组列表
2. 处理当前人员
3. 当前人员处理完成或失败后，记录结果
4. 继续处理下一个人员
5. 全部处理完成后，给出汇总结果

### 4.2 与现有“去处理”流程关系

智能合并的处理入口虽然不同，但每个人的处理流程应尽量复用现有“去处理”能力中已实现的逻辑，包括：

- 根据身份证查询体检记录
- 查询体检详情
- 生成合并项
- 形成最终保存数据
- 调用保存接口

差异仅在于：

- 不弹出“选择两份体检数据”供用户勾选
- 不要求用户手工选择主数据
- 不要求用户手工解决冲突
- 由系统自动完成主辅判定、冲突裁决和保存

### 4.3 执行前风控校验

智能合并开始前必须先进行以下校验：

1. 当前左侧列表对应的筛选日期范围跨度不能大于 180 天
2. 若日期跨度大于 180 天，则本次不允许启动智能合并
3. 页面需弹出明确提示，告知用户缩小筛选范围后再执行

日期跨度说明：

- 按筛选开始日期与结束日期自然日计算
- 超过 180 天即视为不允许执行

## 5. 多条重复体检的处理规则

### 5.1 两条重复体检

若某人员仅有 2 条重复体检，则按以下流程：

1. 查询两条体检详情
2. 自动判定主数据与辅数据
3. 按自动合并规则生成结果
4. 保存主数据
5. 如启用“自动删除辅数据”，则保存成功后删除辅数据

### 5.2 大于两条重复体检

若某人员重复体检数量大于 2，则按以下规则处理：

1. 先在该人员全部重复体检中自动选出 1 条主数据
2. 剩余所有体检均视为辅数据
3. 按顺序将每一条辅数据逐条合并到当前主数据
4. 每合并完一条辅数据，都应基于“最新主数据结果”继续和下一条辅数据合并
5. 全部辅数据合并完成后，最终只保存 1 条主数据结果
6. 若启用自动删除，则在最终保存成功后，按顺序删除所有辅数据

说明：

- 整个过程中最终保留 1 条主数据
- 其余辅数据均属于“待删除候选数据”
- 若删除失败，需要记录到删除失败清单中

### 5.3 大于三条重复体检的跳过规则

若某人员重复体检数量大于 3，则该人员必须直接跳过，不进入自动合并流程。

跳过原因：

- 防止用户由于筛选条件过宽、筛选错误或历史数据跨度过大
- 将原本正常存在的多次体检错误识别为“重复体检”
- 进而导致错误合并历史正常数据

跳过后的处理要求：

- 该人员不查询详情
- 不执行自动判主
- 不执行保存
- 不执行删除
- 在结果中记录为“已跳过”
- 在明细中展示跳过原因：`重复体检数量大于3，存在误合并风险`

说明：

- `2 条`：允许自动合并
- `3 条`：允许自动合并，按“1条主数据 + 2条辅数据”顺序处理
- `大于 3 条`：禁止自动合并，直接跳过

## 6. 自动选择主数据规则

### 6.1 主数据判定目标

系统不再让用户手工选择主数据，而是基于体检详情内容自动判定哪条记录作为主数据。

### 6.2 第一优先级：非空节点数量

优先比较每条体检详情中“非空节点数量”，非空节点数量更多的一条作为主数据。

非空节点定义：

- `value != null`
- `value != ''`
- `value != []`

统计口径说明：

- 仅统计 `serviceData` 顶层节点
- 不展开数组内部元素单独计数
- 不展开对象数组内部字段单独计数

### 6.3 第二优先级：体检日期

若两条体检的非空节点数量一致，则比较体检日期，日期更近的一条作为主数据。

体检日期获取规则：

- 在体检详情节点中查找：
  - `parentId = 200261`
  - `csvId = 300702`
- 取该节点的 `value` 作为体检日期

示例：

```json
{
  "csvId": 300702,
  "parentId": 200261,
  "value": "2024-01-11"
}
```

### 6.4 体检日期格式兼容规则

体检日期字段需要兼容以下两种格式：

- `2026-04-10`
- `2026-04-10 00:00:00`

比较时仅精确到“年月日”，不比较时分秒。

### 6.5 日期为空、缺失、非法时的规则

若非空节点数量一致，且需要比较体检日期时，按以下规则处理：

1. 若一条体检日期有效，另一条为空、缺失或非法，则选择日期有效的那一条作为主数据
2. 若两条体检日期都为空、缺失或非法，则选择第一条作为主数据

### 6.6 全部条件相同的兜底规则

若两条体检满足以下条件：

- 非空节点数量一致
- 体检日期一致

则选择第一条作为主数据。

## 7. 自动合并规则

### 7.1 总体原则

主数据自动选定后，辅数据按现有已实现的体检合并规则自动并入主数据。

### 7.2 冲突处理规则

若主辅两边某节点值冲突，则：

- 直接以主数据为准
- 舍弃辅数据冲突值
- 不再弹出人工决策界面

### 7.3 非冲突处理规则

沿用现有规则：

- 两边都不存在：忽略
- 仅主数据存在：保留主数据
- 仅辅数据存在：补充到主数据
- 两边值相等：保留主数据

### 7.4 多辅数据连续合并规则

当某人员存在 3 条及以上重复体检时：

- 第一轮：主数据 + 辅数据1 -> 生成新主数据
- 第二轮：新主数据 + 辅数据2 -> 再生成新主数据
- 依次循环，直到全部辅数据合并完成

## 8. 保存与删除规则

### 8.1 保存规则

每个人员在完成自动合并后，调用现有保存接口保存最终主数据。

要求：

- 保存参数结构与现有手工合并保存保持一致
- 若保存失败，则该人员本次处理记为失败
- 保存失败后，不执行删除辅数据

### 8.2 删除规则

若用户在策略弹框中勾选“合并保存成功后，自动删除辅数据”，则：

1. 当前人员保存成功后
2. 按顺序调用删除辅数据接口
3. 删除完成后再进入下一个人员

### 8.3 删除失败处理

若出现“保存成功，但删除失败”，则该人员应记为：

- `部分成功`

同时：

- 需要记录删除失败的辅数据
- 在最终处理结果中单独提示用户
- 提醒用户这些辅数据需要后续人工手动处理

说明：

- 删除失败不应中断整批智能合并流程
- 删除失败后仍继续处理下一个人员

### 8.4 删除接口预留要求

当前删除接口可先使用假接口预留，建议包含：

- 删除目标体检记录 ID
- 删除执行结果
- 删除失败原因

建议定义统一返回结构，例如：

```json
{
  "success": false,
  "message": "删除失败，接口未接入"
}
```

## 9. 异常处理与容错

### 9.1 单人失败不影响整体

若某个人在以下任一环节失败：

- 查询体检列表失败
- 查询体检详情失败
- 自动判主失败
- 合并失败
- 保存失败
- 删除失败

或触发以下跳过条件：

- 重复体检数量大于 3

系统都需要记录结果，并继续处理下一个人员。

### 9.2 失败记录内容

建议失败记录至少包含：

- 姓名
- 身份证号
- 失败环节
- 失败原因
- 失败时间

### 9.3 跳过记录内容

建议跳过记录至少包含：

- 姓名
- 身份证号
- 重复体检数量
- 跳过原因
- 跳过时间

### 9.4 删除失败记录内容

建议删除失败记录至少包含：

- 姓名
- 身份证号
- 主数据记录标识
- 删除失败的辅数据记录标识
- 失败原因

## 10. 进度展示要求

### 10.1 执行中展示

智能合并开始后，页面应实时展示：

- 当前总任务数
- 当前已完成数
- 当前成功数
- 当前失败数
- 当前跳过数
- 当前删除失败数
- 当前正在处理的人员
- 当前步骤
- 当前执行状态

### 10.2 执行完成展示

全部处理完成后，需要给出结果汇总，至少包含：

- 总人数
- 成功人数
- 失败人数
- 跳过人数
- 部分成功人数
- 删除失败条数

并在页面中明确提醒：

- 哪些人员合并失败
- 哪些人员被跳过及原因
- 哪些辅数据删除失败，需要人工处理

### 10.3 用户体验建议

建议增加以下交互能力：

- 执行中禁止重复点击“智能合并”
- 执行中展示 loading/处理中状态
- 执行完成后允许用户查看明细
- 执行完成后允许用户再次发起新一轮智能合并
- 因日期跨度大于 180 天被禁止执行时，应给出明确提示，不进入执行态

## 11. 与现有手工合并能力的关系

新增“智能合并”后，现有手工合并流程仍需保留，二者并存：

- `去处理`：适用于人工逐条判断、人工处理冲突
- `智能合并`：适用于按规则批量自动处理

要求：

- 不破坏现有手工合并逻辑
- 两套流程尽量复用同一套数据查询、合并计算、保存能力
- 自动合并逻辑建议封装为独立服务或控制器，避免页面代码膨胀

## 12. 实现建议

### 12.1 架构建议

建议将智能合并拆分为独立模块，避免将复杂流程直接堆积在页面中：

- 智能合并策略模型
- 智能合并执行控制器
- 主数据自动判定服务
- 合并流程编排服务
- 删除服务
- 进度状态模型

### 12.2 复用建议

应尽量复用现有能力：

- 查询重复体检列表服务
- 查询体检详情服务
- 合并项生成规则
- 保存接口逻辑

### 12.3 风险点

实现时需要特别注意以下风险：

- 同一人员存在 3 条以上重复体检时，连续合并后的主数据上下文更新是否正确
- 自动判主时，体检日期节点可能缺失，必须有稳妥兜底
- 删除逻辑不能影响保存成功结果
- 批量执行时页面状态、日志、失败清单不能丢失
- 自动流程不应与现有手工流程状态互相污染

## 13. 验收标准

满足以下条件可认为需求完成：

1. 左侧列表底部已新增“智能合并”按钮
2. 点击按钮后会弹出“合并策略”弹框
3. 用户可选择是否在合并完成后自动删除辅数据
4. 系统可按左侧列表顺序逐人串行自动处理
5. 2 条重复体检可自动判定主辅并完成保存
6. 大于 2 条重复体检可自动选 1 条主数据，并将其余数据逐条合并
7. 主数据自动选择规则符合本文定义
8. 自动合并时冲突默认以主数据为准
9. 保存成功后可按策略自动删除辅数据
10. 删除失败会单独记录并提醒人工处理
11. 某个人处理失败不会中断整个批量任务
12. 某个人重复体检数量大于 3 时会被直接跳过，并记录跳过原因
13. 当前筛选日期跨度大于 180 天时，智能合并不可启动，并给出明确提醒
14. 页面可展示处理进度和最终结果汇总，且包含跳过数量与跳过明细
