# 体检表单冲突合并功能需求文档

## 1. 概述
本功能旨在解决两份体检数据（JSON数组）合并时的冲突处理问题。用户选择主数据后，系统自动将辅数据合并到主数据，对无冲突数据自动处理，对有冲突或仅辅数据存在的数据提供界面供用户人工决策，最终生成一份清洗后的完整数据。

## 2. 数据基础

### 2.1 输入数据
*   **输入对象**：两个 JSON 数组，分别标记为 **A** 和 **B**（选择主数据后确定主辅关系）。
*   **元素结构**：
    ```json
    {
      "csvId": Int,          // 节点ID
      "parentId": Int,       // 父节点ID
      "value": any,          // 值类型：String, Number, Array<Int>, Array<String>, "", []
      "csvName": String      // (显示用) 节点名称
    }
    ```

### 2.2 唯一标识
*   **Key**: `(parentId + csvId)`
*   组合唯一确定一个业务节点，用于两份数据间的对齐比较。

### 2.3 节点存在性定义
*   **存在**：数组中包含该节点 **且** `value` 不为 `""` (空字符串) 且不为 `[]` (空数组)。
*   **不存在**：数组中无此节点，**或** `value` 为 `""` 或 `[]`。
*   **处理原则**：空值 (`""`, `[]`) 视为无效数据，不参与具体的数值比较，直接视为"不存在"。

## 3. 页面流程

```
搜索体检 → 选择2个体检（必须选2个，否则提示） → 查询详情 → 选择主/辅数据 → 自动合并显示 → 手动解决冲突 → 保存（打印JSON）
```

### 3.1 搜索栏
- 日期范围选择器（开始日期、结束日期）
- 身份证号输入框
- 搜索按钮

### 3.2 体检选择
- 点击搜索后弹出体检列表供用户选择
- **必须选择2个体检**，未选或只选1个时提交给予提示
- 选择2个后查询详情数据

### 3.3 主辅数据选择
- 查询完成后页面刷新，以**卡片形式**展示体检A的日期、体检B的日期
- 卡片支持**单选**，选中后代表以对应数据为**主数据**，另一个为**辅数据**
- 例如：选中了A为主数据，则后续要把B的数据合并到A

## 4. 合并规则

### 4.1 数据分类
根据主辅数据的存在状态，将所有节点分为四类：

| 类型 | 定义 | 处理方式 | 卡片色调 |
|------|------|----------|----------|
| **两边都不存在** | 主辅两边均无此节点，或 value 均为无效空值 | **隐藏**，不显示在界面上 | - |
| **仅主数据存在** | 主数据有效，辅数据不存在（或为空） | **自动保留**，无需用户操作 | 白色 |
| **仅辅数据存在** | 辅数据有效，主数据不存在（或为空） | **自动保留**，无需用户操作 | **绿色**（代表新增） |
| **两边都存在** | 主辅两边都有效（非空） | 需进一步对比 value 值 | - |

### 4.2 两边都存在的情况

#### A. 值相等（无冲突）
*   **判定规则**：
    *   **弱类型相等**：字符串 "123" 与数字 `123` 视为相等。
    *   **数组无序相等**：`[400591, 400592]` 与 `[400592, 400591]` 视为相等。
*   **处理**：**自动保留主数据值**，用户无需操作。
*   **显示**：白色卡片，显示节点名称+值。

#### B. 值不相等（已冲突）
*   **定义**：两边值不一致。
*   **处理**：用户必须选择 **使用主数据** 或 **使用辅数据** 来解决冲突。
*   **卡片色调**：**红色**（未解决），解决后变为**绿色**。
*   **支持修改**：用户后续可以修改选择（比如先前选择主数据，后面可以改选辅数据）。

## 5. 界面显示规则

### 5.1 布局方式
**紧凑卡片列表 + 筛选栏**

考虑到数据量约50项，采用紧凑设计以在一屏内显示更多数据。

### 5.2 筛选栏
顶部显示筛选标签，快速过滤查看不同类型的数据：

| 筛选标签 | 显示内容 |
|----------|----------|
| **全部** | 显示所有节点（除两边都不存在的） |
| **冲突** | 仅显示冲突项（未解决红色+已解决绿色） |
| **新增** | 仅显示辅数据独有项（绿色） |
| **主数据** | 仅显示主数据独有+两边相等项（白色） |

### 5.3 卡片设计（紧凑化）

#### 主数据独有/两边相等（白色卡片）
```
┌─────────────────────────────────────┐
│ 节点名称                值: xxx      │
└─────────────────────────────────────┘
```

#### 辅数据独有（绿色卡片）
```
┌─────────────────────────────────────┐  ← 绿色边框/浅绿背景
│ [新增] 节点名称         值: yyy      │
└─────────────────────────────────────┘
```

#### 冲突项（红色卡片 → 解决后绿色）
未解决状态：
```
┌─────────────────────────────────────┐  ← 红色边框/浅红背景
│ [冲突] 节点名称         [待解决]    │
│ 主: xxx                    辅: yyy  │  ← 紧凑左右对比或上下两行
└─────────────────────────────────────┘
```

已解决状态：
```
┌─────────────────────────────────────┐  ← 绿色边框
│ [冲突] 节点名称         [✓ 已选主]  │
│ 值: xxx（来自主数据）               │
└─────────────────────────────────────┘
```

### 5.4 选项值映射显示
*   **场景**：体检项的 `value` 可能是普通字符串、数字、枚举值、枚举数组、`枚举值:自由文本` 的混合格式，或“对象数组”结构。对于直接显示数字 ID 不友好的情况，需要转换为对应中文名称。
*   **参考文件**：`assets/data/physical_item_enum_value.json`
*   **核心原则**：
    1.  先使用当前节点的 `csvId` 作为 key，在 `physical_item_enum_value.json` 中查找对应枚举定义。
    2.  只有当该 `csvId` 能找到配置项时，才进入枚举映射流程；否则直接显示原始 `value`。
    3.  枚举判断只针对以下格式的编码：
        *   以 `300` 或 `400` 开头
        *   总长度为 6 位
        *   能在当前 `csvId` 对应的枚举列表中找到匹配 `value`
    4.  满足上述条件的编码才做转换；否则保持原值，不做误转换。
    5.  当 `value` 为对象数组时，对象内每个字段都应以“字段自身的 key”作为枚举映射 key 单独处理，而不是继续使用外层节点的 `csvId`。
*   **值类型处理规则**：
    1.  **普通字符串 / 普通数字**
        *   示例：`"正常"`、`123`
        *   处理：直接显示原值
    2.  **单个枚举值**
        *   示例：
            ```json
            {
              "csvId": 300506,
              "parentId": 123456,
              "value": "300141"
            }
            ```
        *   处理：
            *   用 `csvId=300506` 去 `physical_item_enum_value.json` 查找枚举列表
            *   在该列表中找到 `value=300141` 对应的 `label=吸烟状况`
            *   页面显示：`吸烟状况`
    3.  **枚举数组**
        *   示例：
            ```json
            {
              "csvId": 300506,
              "parentId": 123456,
              "value": ["300141", "300143"]
            }
            ```
        *   处理：
            *   数组中的每一项都按当前 `csvId` 对应的枚举表分别转换
            *   页面显示：`吸烟状况, 开始吸烟年龄`
    4.  **枚举值:自由文本**
        *   示例：
            ```json
            {
              "csvId": 300506,
              "parentId": 123456,
              "value": "300141:张三他不吸烟"
            }
            ```
        *   处理：
            *   先按第一个 `:` 拆分为 `枚举编码` 和 `补充内容`
            *   如果前半段是合法枚举编码，且能在当前 `csvId` 的枚举表中找到映射，则只转换前半段
            *   页面显示：`吸烟状况:张三他不吸烟`
        *   说明：
            *   仅拆分第一个 `:`
            *   冒号后的内容原样保留，不参与枚举转换
    5.  **对象数组**
        *   场景说明：
            *   某些节点的 `value` 不是基础类型，也不是普通数组，而是 `List<Map<String, dynamic>>` 结构。
            *   这种结构常见于“用药记录”等复合数据，一个对象内每个字段 key 本身就代表一个子项 `csvId`。
        *   示例：
            ```json
            {
              "csvId": 300401,
              "parentId": 123456,
              "value": [
                {
                  "300363": "400591",
                  "300402": "吲达帕胺片",
                  "300403": "口服",
                  "300406": "5:月",
                  "300764": "2.5mg",
                  "301002": "每日一次"
                }
              ]
            }
            ```
        *   处理规则：
            *   `value` 是数组时，先逐个遍历数组元素；如果元素是对象，再逐个遍历对象中的字段。
            *   对象字段如 `300363`、`300402`、`300403`、`300406`、`300764`、`301002`，每个字段 key 都视为独立的 `csvId`。
            *   每个字段值都按“当前字段 key 对应的枚举表”做提取和转换，不使用外层节点的 `csvId`。
            *   如果字段值是单个枚举编码，则按该字段 key 对应的枚举映射转中文。
            *   如果字段值是 `枚举值:自由文本`，则仍按“先拆前半段编码，再仅转换前半段”的规则处理。
            *   如果字段值不是合法枚举编码，或当前字段 key 在 `physical_item_enum_value.json` 中无配置，则保持原值显示。
        *   示例说明：
            *   `300363: 400591`
            *   以 `300363` 作为映射 key 去枚举表查找 `400591` 对应中文名称后显示。
            *   `300402: 吲达帕胺片`
            *   由于值不是合法枚举编码，直接显示 `吲达帕胺片`。
            *   `300403: 口服`
            *   由于值不是合法枚举编码，直接显示 `口服`。
            *   `300406: 5:月`
            *   虽然包含 `:`，但前半段 `5` 不是合法枚举编码，因此整段按原值显示 `5:月`。
            *   `300764: 2.5mg`
            *   直接显示原值。
            *   `301002: 每日一次`
            *   直接显示原值。
        *   页面展示建议：
            *   对象数组应按“子项名称: 显示值”的形式展开展示，便于人工比对。
            *   子项名称优先使用字段 key 对应的节点名称映射；子项值再按本节规则做枚举转换。
    6.  **其他类型**
        *   示例：无法识别的字符串、对象、混合数组、空值等
        *   处理：按原始值显示，避免错误映射
*   **兜底规则**：
    1.  `csvId` 在 `physical_item_enum_value.json` 中不存在：显示原始值
    2.  `value` 看起来像编码，但不满足“`300/400` 开头 + 6 位 + 当前枚举表可匹配”：显示原始值
    3.  数组中部分元素可映射、部分不可映射：可映射项转中文，不可映射项保留原值
    4.  `枚举值:自由文本` 中前半段不可映射：整段原样显示
    5.  对象数组中部分字段可映射、部分不可映射：逐字段独立处理，可映射项转中文，不可映射项保留原值
    6.  对象数组中的字段 key 若不是合法 `csvId`，则该字段按普通对象字段处理，值原样显示
*   **实现建议**：
    1.  不再依赖 `(parentId + csvId)` 做枚举值映射判断，枚举值显示仅依赖当前节点的 `csvId`
    2.  节点名称仍可继续由 `csv_name_mapping.json` 或接口返回的 `csvName` 提供
    3.  当 `value` 为对象数组时，内部字段名称可继续由 `csv_name_mapping.json` 或接口返回的节点名称提供，字段值映射则改为依赖“字段 key 对应的 `csvId`”
    4.  本规则适用于所有体检项显示场景，包括主数据值、辅数据值、冲突对比值、合并结果值

## 6. 排序规则

列表按以下优先级排序（从高到低）：

1. **冲突项**（未解决红色在前，已解决绿色在后）
2. **辅数据独有**（绿色，代表新增）
3. **主数据独有 + 两边相等**（白色）

## 补充：重复体检档案筛选

### 功能背景
- 在进行体检合并前，系统需要先批量筛出当前机构内的重复体检档案。
- 筛出的重复数据展示在页面最左侧，方便用户直接复制身份证号，并在右侧快速搜索两份体检进行合并。

### 触发时机
- 用户在 `physical_merge_page.dart` 页面登录成功后，且已获取到有效 `token`，系统自动弹出“筛选重复体检档案”弹框。
- 如果用户已关闭弹框，也可通过左侧列表顶部的“重新筛选”按钮再次打开。
- 用户点击弹框“取消”时，不清空页面左侧已加载的重复档案数据，继续保留上一次确认后的结果。

### 查询接口
- 查询地址固定为：
  `http://bmg.2woniu.cn/bmanage/publichealth/service/record/1/list`
- 请求方式：`GET`
- 固定参数：
  - `unRegionCode=0`
  - `pageNo=1`
  - `nodeId=1`
  - `isUpload=2`
- 动态参数：
  - `peStart`：开始日期
  - `peEnd`：结束日期
  - `keyword`：关键词，支持用户名和身份证号搜索
  - `pageSize`：分页条数
  - `token`：登录成功后获取

### 弹框筛选条件
- 开始日期
- 结束日期
- 关键词
- 分页条数
- 分页条数默认值为 `800`
- 分页条数下拉可选值：`20`、`80`、`100`、`400`、`800`、`1000`
- 弹框打开后不自动发起查询
- 每次查询都必须由用户点击“搜索”按钮后才开始执行

### 数据处理规则
- 接口返回成功条件：`status = 0`
- 接口返回的 `results` 需要在前端按身份证号 `idCard` 分组
- 仅展示存在重复的数据：
  - 若某个身份证分组下只有 1 条记录，则该组直接剔除
  - 若某个身份证分组下大于 1 条记录，则视为重复组并展示
- 页面需要显示当前共筛出多少组重复数据，组数口径为“按身份证号去重后的分组数量”
- 每组内数据按体检日期倒序展示，便于优先查看较新的体检记录

### 弹框展示规则
- 弹框标题为“筛选重复体检档案”
- 顶部展示当前筛出的重复组数量
- 初次打开弹框且尚未点击搜索时，列表区域显示“请设置条件后点击“搜索”开始查询”
- 每个身份证号为一个分组，同组数据需要在视觉上能明显区分为同一组
- 每组支持展示多条体检数据
- 每条数据展示以下字段：
  - 体检日期
  - 姓名
  - 性别
  - 年龄
  - 身份证号

## 补充：智能合并

### 功能目标
- 在保留现有“去处理”手工合并能力的前提下，新增一套“智能合并”批量处理能力。
- 智能合并适用于重复体检较明确、数据时间跨度较小的场景，用于减少人工逐条选择主辅数据和逐条处理冲突的成本。

### 页面入口
- 在左侧“重复体检档案”列表底部新增 `智能合并` 按钮。
- 点击按钮后，不直接执行，而是先弹出“智能合并策略”弹框。

### 智能合并策略弹框
- 弹框中需要向用户说明以下规则：
  - 系统按左侧重复档案列表顺序逐人串行处理
  - 每个人处理完成后才继续处理下一个
  - 主数据由系统自动判定
  - 冲突项自动以主数据为准
  - 某个人处理失败不会中断整体流程
- 弹框中提供一个开关项：
  - `合并保存成功后自动删除辅数据`
- 删除接口当前允许先用假接口预留，后续再替换为真实接口。

### 风控限制
- 若当前筛选日期跨度 **大于 180 天**，则禁止启动智能合并。
- 禁止时需要明确提醒用户：筛选时间跨度过大，存在把历史正常体检错误合并的风险。
- 若某个人员的重复体检数量 **大于 3 条**，则该人员必须直接跳过，不参与自动合并。
- 跳过原因需要记录为：`重复体检数量大于3，存在误合并风险`。

### 智能合并处理流程
- 系统遍历左侧重复档案列表，按顺序逐个处理。
- 每个人的处理流程与用户点击“去处理”后的主流程保持一致，复用已有的查询详情、构建合并项、保存等能力。
- 区别在于：
  - 不需要用户手工选择 2 条体检
  - 不需要用户手工选择主辅数据
  - 不需要用户手工处理冲突
  - 全流程自动完成

### 多条重复体检的处理规则
- 若某个人员只有 `2` 条重复体检：
  - 自动选择主数据和辅数据
  - 自动合并
  - 自动保存
- 若某个人员有 `3` 条重复体检：
  - 自动从 3 条中选出 1 条主数据
  - 其余 2 条按顺序逐条并入主数据
  - 最终只保存 1 条主数据结果
- 若某个人员重复体检数量 `> 3`：
  - 直接跳过
  - 不查询详情
  - 不执行保存
  - 不执行删除

### 自动选择主数据规则
- 第一优先级：比较体检详情中“非空节点数量”，非空数量更多的一条作为主数据。
- 非空节点统计口径：
  - `value != null`
  - `value != ''`
  - `value != []`
  - 仅统计 `serviceData` 顶层节点
  - 不展开数组内部元素单独计数
  - 不展开对象数组内部字段单独计数
- 第二优先级：若非空节点数量一致，则比较体检日期，日期更近的一条作为主数据。
- 体检日期取值规则：
  - 通过 `parentId = 200261` 且 `csvId = 300702` 唯一定位节点
  - 取该节点的 `value` 作为体检日期
- 体检日期格式需要兼容：
  - `yyyy-MM-dd`
  - `yyyy-MM-dd HH:mm:ss`
- 日期比较仅比较“年月日”，不比较时分秒。
- 若一条日期有效，另一条日期为空、缺失或非法，则选择日期有效的一条作为主数据。
- 若两条日期都为空、缺失或非法，则选择第一条作为主数据。
- 若两条数据的非空数量一致，且体检日期也一致，则选择第一条作为主数据。

### 自动合并规则
- 主数据确定后，辅数据按现有体检合并规则自动并入主数据。
- 冲突项默认以主数据为准，辅数据冲突值直接舍弃。
- 无冲突项继续沿用原有规则：
  - 仅主数据存在：保留主数据
  - 仅辅数据存在：补充到主数据
  - 两边值相等：保留主数据
  - 两边都不存在：忽略
- 对于 3 条体检的情况，采用“累积合并”方式：
  - 第一轮：主数据 + 辅数据1
  - 第二轮：上一轮结果 + 辅数据2

### 保存与删除
- 自动合并完成后，调用现有保存接口保存最终主数据。
- 若用户勾选“自动删除辅数据”，则保存成功后继续按顺序调用删除接口删除辅数据。
- 当前删除接口允许先使用假接口预留。
- 若保存成功但删除失败，则该人员状态记为：
  - `部分成功`
- 删除失败需要单独记录，便于最终提醒用户后续人工处理。
- 删除失败不会中断整批智能合并流程。

### 异常处理
- 某个人处理失败时，需要记录失败环节和失败原因，并继续处理下一个人员。
- 单人失败不允许中断整个智能合并批次。
- 失败记录建议至少包含：
  - 姓名
  - 身份证号
  - 失败阶段
  - 失败原因

### 进度展示
- 页面需要新增“智能合并进度”展示区域。
- 建议至少展示：
  - 总人数
  - 已处理人数
  - 成功人数
  - 部分成功人数
  - 失败人数
  - 跳过人数
  - 当前处理人员
  - 当前步骤
- 全部处理完成后，需要支持查看结果明细。
- 明细中应能区分：
  - 成功人员
  - 部分成功人员
  - 失败人员
  - 跳过人员
  - 删除失败记录

### 与手工合并的关系
- 手工合并和智能合并需要并存：
  - `去处理`：用于人工逐条确认
  - `智能合并`：用于按规则批量自动执行
- 智能合并不能破坏原有手工合并流程。
- 两套流程应尽量复用同一套详情查询、合并计算和保存能力。
- 弹框底部提供“取消”和“确定”按钮
- 当未查询到任何重复数据时，点击“确定”按钮需要提示并阻断关闭弹框，不能将空结果带回主页面
- 点击“确定”后，将当前弹框中展示的重复组数据带回主页面左侧列表

### 主页面左侧列表展示
- 左侧列表需要顶端和底端对齐页面可用高度，形成独立滚动区域
- 左侧列表顶部展示：
  - 标题“重复体检档案”
  - “重新筛选”按钮
  - “共 X 组数据”文案
  - 当前生效的筛选条件摘要
- 左侧列表按身份证号去重后展示，每个身份证号仅显示 1 行
- 每行至少展示以下信息：
  - 身份证号
  - 重复条数
- 示例：
  - 身份证号 `510802195003205394` 若存在 3 条记录，则左侧仅展示 1 行：`510802195003205394`、`3条`
- 身份证号需要支持鼠标选中复制，方便右侧快速检索与合并

### 筛选条件状态同步
- 用户在重复体检筛选弹框中点击“确定”后，需要将以下筛选条件同步回页面状态：
  - 开始日期
  - 结束日期
  - 关键词
  - 分页条数
- 页面再次点击“重新筛选”时，应默认带出上一次确认的筛选条件
- 重复体检筛选的关键词与右侧体检合并搜索输入框相互独立，不能互相覆盖

### 7.3 筛选操作
- 点击筛选标签快速过滤列表
- 默认显示"全部"

### 7.4 保存操作
- 页面底部有**保存按钮**
- 点击时校验所有冲突是否已解决
- 如有未解决的冲突，给予提示不予提交
- 校验通过后打印合并后的 JSON（后续扩展为真实保存）

## 8. 样式规范

### 8.1 颜色定义
| 类型 | 卡片边框/背景 | 状态标识 |
|------|--------------|----------|
| 主数据独有 | 白色/灰色50 | - |
| 辅数据独有（新增） | 绿色300/绿色50 | [新增] 标签 |

### 8.2 设计约束
- **单列表**：页面只显示一个合并后的列表
- **卡片紧凑**：减小内边距（8-12px），节点名称和值尽量在同一行
- **筛选便捷**：顶部固定筛选栏，快速定位

### 8.3 页面风格
- 蓝色风格为主（顶部搜索栏、按钮等）

## 9. 最终输出

### 9.1 输出内容
合并后的 JSON 数组包含：
1.  **主数据独有项**：原样保留
2.  **两边相等项**：保留主数据值
3.  **辅数据独有项**：保留辅数据值（新增）
4.  **用户解决冲突项**：用户选择的解决结果（主数据值或辅数据值）

### 9.2 排除内容
1.  **两边都不存在的节点**
2.  **用户删除的冲突项**（如支持删除功能）

### 9.3 输出方式
- 保存时打印合并后的 JSON 到控制台（后续扩展为真实保存）
